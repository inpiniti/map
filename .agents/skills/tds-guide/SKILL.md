# 앱인토스(TDS) 디자인 가이드

앱인토스 미니앱 개발 및 디자인 시 준수해야 할 가이드라인 모음이에요.

## 1. 브랜드 로고 / 이름 / 컬러

브랜드 로고, 브랜드 이름, 브랜드 컬러를 토스와 앱인토스 화면 곳곳에 노출해 사용자가 브랜드를 분명하게 인지할 수 있게 해요.

### 브랜드 로고

- 크기: **600×600px** 각진 정사각형 (모서리 둥근 형태 사용 불가)
- 로고 뒤에 배경 필수 (라이트/다크 모드 모두 잘 보이는 배경 색상)
- 로고 자체에 배경 포함된 경우 600×600px 영역에 꽉 차게 배치
- `granite.config.ts`의 `appsInToss` 함수 → `brand.icon` 속성에 로고 링크 입력

### 브랜드 이름

- 특별한 이유 없으면 **한글**로 작성 (예: `토스` O, `Toss` 비권장)
- `granite.config.ts` → `brand.displayName` 속성에 입력

### 브랜드 컬러

- 기존 브랜드 컬러 있으면 그대로 사용
- 없으면 로고에서 가장 많이 사용된 색상 선택
- `granite.config.ts` → `brand.primaryColor` 속성에 `#RRGGBB` 형식으로 입력 (예: `#3182F6`)
- 색 대비 기준 미충족 시 자동 보정됨

## 2. 내비게이션 바

화면 상단에 고정되는 영역으로 앱인토스 전용 컴포넌트 사용.
- 게임 서비스 / 비게임 서비스에 따라 가이드 상이

## 3. 탭바

- 필수 컴포넌트 아님
- 필요 시 토스에서 제공하는 **플로팅 형태** 탭바를 직접 구현
- 탭 개수: **최소 2개, 최대 5개**
- 토스 메인 하단 탭과 형태가 겹치지 않도록 플로팅 형태 유지 필수

## 4. 선택 UI 패턴 (권장)

반복적으로 고르는 항목(예: 실행시간, 티커그룹, AI 모델)은 **화면 내 칩/드롭다운 혼합 대신 하단 바텀시트 선택 패턴으로 통일**해요.

### 권장 방식

- 입력 필드는 읽기 전용 선택 박스 형태로 보여주고, 탭하면 바텀시트가 올라오게 해요.
- 바텀시트 안에서 단일 선택 리스트를 보여주고, 현재 선택값은 체크 아이콘으로 표시해요.
- 선택 즉시 닫히게 구성해 입력 단계를 짧게 유지해요.
- 같은 유형의 선택 항목은 같은 높이/패딩/아이콘(chevron) 패턴을 재사용해요.

### 주의사항

- 서비스 진입 직후 바텀시트를 자동으로 열지 않아요. (다크패턴 정책 준수)
- 닫기 동작(딤 영역 탭, 뒤로가기)을 항상 제공해요.
- 선택지가 많을 때는 기본값을 합리적으로 제안하고, 목록 구조를 유지해 스캔하기 쉽게 만들어요.

## 다크패턴 방지 정책

사용자에게 예측 가능하고 편리한 경험을 제공하기 위한 최소 기준. 아래 사례는 앱인토스 서비스로 출시 불가.

### 금지 패턴

1. **서비스 진입 직후 바텀시트 노출** — 광고성 바텀시트, 알림 동의 바텀시트 등 초기 인터럽트 금지
2. **뒤로 가기 시 바텀시트로 이탈 막기** — 알림 동의 유도 등 의도적 이탈 방지 인터럽트 금지
3. **나갈 수 있는 선택지 없음** — 파트너 CTA 외 거절 불가 구조 금지 (강제적 설계)
4. **예상 못한 순간에 광고 노출** — 아이템 획득 등 사용 흐름 중 전면 광고 돌출 금지
5. **CTA 버튼 라벨 불명확** — 버튼 클릭 후 어떤 화면/행동으로 이어지는지 예측 불가 구조 금지

## UX 라이팅 가이드

토스 앱 보이스톤을 적용한 문구 작성 원칙.

### 1. 해요체 사용

모든 문구에 **해요체** 적용 (상황·맥락 불문)

### 2. 능동적 말하기

- 됐어요 → **했어요**
- '~었' 빼기
- 수동형 동사 → 능동형 동사로 변환

### 3. 긍정적 말하기

- 안 돼요, 없어요 (X) → ~하면 할 수 있어요 (O)
- 에러 메시지도 긍정형 문장 권장
- 다이얼로그 왼쪽 버튼: **닫기** (취소 사용 금지)

### 4. 캐주얼한 경어

- `~시겠어요?`, `~시나요?`, `~께` 등 과도한 경어 금지
- `계시다` → `있다`
- `여쭈다` → `확인하다, 묻다`
- `께` → `에게`

### 5. '{명사}+{명사}' 쓰지 않기

- 한자어 명사 → 동사 형태로 풀어쓰기
- 풀기 어려우면 `{명사}가 {명사}해서` 형태로 변환

## 그래픽 리소스

> 토스 그래픽 자산은 앱인토스 제휴 환경 내 서비스 운영/홍보 목적으로만 사용 가능

### 제공 리소스 종류

1. **아이콘 & 이모지**: 약 7,000개 이상 (앱빌더/피그마에서 확인)
   - 화면에서 24~40px 크기로 사용
   - 아이콘·이모지 두 개 이상 병렬 조합 금지
   - 앱 로고/썸네일 등 앱 정보에는 사용 불가
2. **스마트폰 목업 파일**: 제공 파일 그대로 사용 (임의 크롭/색상 보정/형태 왜곡 금지)
3. **토스트(AI 이미지 생성 툴)**: 생성 후 토스 그래픽 디자인 팀 승인 필요 (보통 1일 이내)
4. **3D 그래픽/애니메이션**: 토스 제공 모듈 내 리소스만 사용 가능

### 그래픽 사용 원칙

| 원칙 | 설명 |
|------|------|
| 문맥에 맞는 그래픽 | 장식이 아닌 화면 의미 전달 용도 |
| 정보 밀도에 맞는 크기 | 단순 = 작게, 디테일 많음 = 크게 |
| 한 화면에 핵심 그래픽 하나 | 나머지는 보조 그래픽/아이콘으로 |
| 핵심 정보 가리지 않기 | 불필요한 스크롤 방지 |
| 부정적 감정 표현 금지 | 불쾌감, 애원, 호소 표현 = 다크패턴 |
| 장식적 이펙트 금지 | 파티클, 과한 그라데이션 등 금지 |
| 상황에 맞는 그래픽 | 오류 아닌데 느낌표, 대기 불필요한데 로딩 = 금지 |

### 파트너사 그래픽 제작 시 유의사항

- 토스 스타일: 단순·명료·깨끗한 디지털 그래픽 (손그림·만화체·서정적 화풍 금지)
- 고화질 제작 필수 (저해상도, 자잘한 파티클 금지)
- 다크/라이트 모드 모두에서 잘 보이는 **중간 명도 색감** 사용
- 그래픽이 텍스트/CTA보다 튀지 않도록 화면 전체 균형 유지
- 밝고 정돈된 긍정적 인상 유지 (모노톤 과도/뿌옇고 칙칙한 인상 금지)

## 해상도 가이드

앱인토스 미니앱 개발 시 해상도 기준.

### 풀스크린 설계 필수 사항 (게임 미니앱)

- 콘텐츠가 화면 완전히 채워야 함
- 웹뷰 여백/반투명 영역 남기지 않기
- 기기 회전 시 비율 깨짐·레터박스 금지
- 노치·카메라 홀·Dynamic Island → Safe Area 처리
- 스케일링으로 인한 에셋 품질 저하 방지

### 해상도 구분

| 구분 | 설명 |
|------|------|
| **논리 해상도** | UI 배치·좌표·게임 로직 기준 / 하나만 선택 |
| **에셋 해상도** | 이미지·배경·캐릭터의 픽셀 밀도 |

### 권장 논리 해상도

- **세로형**: 약 360×640 ~ 420×740
- **가로형**: 약 640×360 ~ 740×420

### 권장 에셋 해상도

- **1x**: 기본 해상도 대응
- **2x**: 고해상도 디바이스 대응
- 3x: 그래픽 품질이 특히 중요한 경우에만 선택적 추가

> 대부분의 미니앱은 1x + 2x 구성만으로 충분. 과도한 해상도 그룹 → 메모리 증가·로딩 지연·유지보수 비용 증가

### 실제 트래픽 데이터 기반

- 전체 트래픽 **70~80%**: 가로 약 800~900, 세로 약 360~420 viewport 집중
- 나머지: 다양한 해상도 롱테일 분포

### 테스트 기기 권장 (3~5종)

- 화면 비율이 다른 기기 2~3종
- Safe Area가 크게 적용되는 기기 1종
- 비교적 작은 화면 크기 기기 1종

### 권장하지 않는 방식

- 디바이스 모델별로 다른 논리 해상도 사용
- 실제 픽셀 해상도 기준으로 UI·좌표 계산
- 필요 이상으로 많은 에셋 해상도 그룹 운영
- 해상도마다 다른 UI 레이아웃 유지
